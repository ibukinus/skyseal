# 専用ページの本文取得はサービスAPIを経由する

## ステータス

accepted（2026-07-04）

## コンテキスト

MVP要件（[requirements/mvp.md](../requirements/mvp.md) 6.7）では、専用ページの初期HTMLにネタバレ本文を含めず、ページ読み込み後に取得して表示すると定めている。この「取得」の経路には2つの選択肢がある。

1. ブラウザから投稿者PDSへ直接アクセスして本文レコードを取得する
2. サービスが提供する本文取得APIを経由して取得する

投稿者のPDSはDIDごとに異なる任意のホストであるため、経路1ではCSPの `connect-src` を実質ワイルドカードにせざるを得ず、CSP要件（7.3）が形骸化する。また、要件7.3が求める本文取得APIへのレート制限は、経路1では適用対象が存在しない。閲覧者のIPアドレスが投稿者PDSへ直接渡る点も経路1の副作用となる。

一方、経路2ではネタバレ本文がサービスのリクエスト処理を通過するため、本文の非永続化・非ログ出力（7.1、7.2）をサービス実装で厳守する必要がある。

## 決定

本文取得はサービス提供の本文取得APIを経由する（経路2）。ブラウザから投稿者PDSへ直接アクセスしない。

* 本文取得APIは、DIDから投稿者の現在のPDSを解決し、対象レコードを取得して返す。
* 本文はリクエスト処理中のみ保持し、永続化・キャッシュ・ログ出力を行わない。
* 本文取得APIにはレート制限を設ける。
* CSPの `connect-src` はサービス自身のオリジンに限定する。

## 検討した代替案

**ブラウザから投稿者PDSへ直接取得する（経路1）**

* 利点：ネタバレ本文がサービス側を一切通過せず、データ最小化の観点で最も強い。サービス側の負荷も小さい。
* 欠点：`connect-src` を実質ワイルドカードにする必要がありCSPが弱まる。サービス側でレート制限やエラーハンドリングを統一できない。閲覧者IPが投稿者PDSに露出する。PDSごとのCORS挙動差に表示品質が左右される。

## 結果

* 要件7.1（本文の非出力先リスト）と7.2（リクエスト処理中のみの保持）が本文取得APIの実装制約として適用される。
* 要件7.3のレート制限は本文取得APIに対して実装する。
* 運営者による表示停止（6.9）は、本文取得APIの応答制御として一元的に実装できる。
